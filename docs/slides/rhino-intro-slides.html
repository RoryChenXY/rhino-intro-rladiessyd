Logic vs View Separation

Logic (app/logic/)

# Pure R functions
# No Shiny dependencies
# Testable and reusable

box::use(
  dplyr[...],
)

#' @export
process_data <- function(df) {
  df |>
    filter(year > 2020) |>
    summarise(total = sum(value))
}

View (app/view/)

# Shiny modules
# UI and reactive logic

box::use(
  shiny[...],
  app/logic/data[process_data],
)

#' @export
ui <- function(id) {
  # Module UI
}

#' @export  
server <- function(id) {
  # Module server
}

Key Features Deep Dive

{box} Modules

# Each R file is a module with its own namespace
box::use(
  dplyr,  # Functions available as dplyr$function()
  shiny[div, moduleServer, NS],  # Import specific functions
)

#' @export
my_function <- function() {
  # Only exported functions are visible outside
}
  • Encapsulation: Each module has its own namespace
  • Explicit imports: No global dependencies
  • Directory structure: Organize modules logically

Best Practices

Module Design

  • Single Responsibility: Each module should have one clear purpose
  • Clear Interfaces: Use @export to define public functions
  • Minimal Dependencies: Import only what you need
  • Logic First: Write business logic before UI components

Development Tips

DO: - Use box::use() for all imports - Separate logic from view - Write tests for logic functions - Use Sass for styling - Leverage CI/CD workflows

DON‚ÄôT: - Use library() calls - Put everything in one file - Skip testing - Modify app.R - Use global variables
